Practica #4

Interbloqueo e Inanicion

Deadlock Inanicion Banquero Planificacion

Concepto general

El interbloqueo es el bloqueo permanente de un conjunto de procesos que compiten por recursos. Para que ocurra deben cumplirse cuatro condiciones: exclusión mutua, retencion y espera, no apropiacion y espera circular.

La inanicion ocurre cuando un proceso espera indefinidamente un evento debido a las politicas de planificacion, aun cuando el sistema sigue avanzando para otros procesos.

4.1 Prevencion

La prevencion consiste en eliminar al menos una de las cuatro condiciones necesarias para que aparezca el interbloqueo. Por ejemplo, imponer un orden total en la solicitud de recursos evita la espera circular.

4.2 Algoritmo del Banquero

Es una estrategia para evitar interbloqueos simulando la asignacion de recursos antes de confirmar una solicitud. El sistema solo concede recursos si el estado resultante sigue siendo seguro, es decir, si existe una secuencia posible de finalizacion para todos los procesos.

4.3 Deteccion y Recuperacion

Cuando no se previene ni evita el interbloqueo, se necesita deteccion y recuperacion. Esto implica identificar ciclos de espera y aplicar medidas como abortar procesos, reiniciarlos, o retirar recursos para romper el bloqueo.

4.4 Codigo: Simulacion conceptual del Algoritmo del Banquero

Codigo Fuente (C)

Basado en la subrutina de Dijkstra.

// Basado en la subrutina de Dijkstra
int dinero = efectivo;
boolean seguro, inseguro[N];

for(i = 1; i <= N; i++) inseguro[i] = TRUE;
for(i = 1; i <= N; i++) {
    if (inseguro[i] && (demanda[i] <= dinero)) {
        inseguro[i] = FALSE;
        dinero = dinero + prestamo[i];
    }
}
if (dinero == capital) seguro = TRUE;
else seguro = FALSE;

El algoritmo marca como inseguros a todos los procesos y trata de encontrar procesos cuya demanda pendiente pueda cubrirse con el dinero disponible. Si un proceso puede terminar, "devuelve" su prestamo y aumenta el dinero disponible para continuar evaluando al resto.

Lo que aprendi y como se podria mejorar

Aprendi a diferenciar entre un estado seguro y uno inseguro, y a comprobar por que no todas las solicitudes deben aceptarse aunque haya recursos disponibles en ese momento.

Se puede mejorar incorporando entradas dinamicas de maximo, asignado y disponible, ademas de mostrar la secuencia segura paso a paso en cada iteracion.
