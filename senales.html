Practica #7

Senales

SIGINT SIGKILL SIGALRM Manejadores

Concepto general

Las señales son interrupciones de software que informan a un proceso sobre eventos asincronos. Un proceso puede ignorar la señal, usar la accion por defecto del sistema (generalmente terminar) o registrar un manejador personalizado para responder de forma controlada.

7.1 Acciones ante una señal

Ante una señal, el proceso puede: ignorarla, aceptar la accion por defecto o ejecutar una rutina propia. Esta flexibilidad es clave para construir programas robustos que limpien recursos y continúen operando en escenarios inesperados.

7.2 Señales comunes

Entre las mas usadas estan SIGINT (Ctrl+C), SIGKILL (terminacion forzosa que no puede ignorarse) y SIGALRM (notificacion de temporizador). Estas señales permiten coordinar control de ejecucion, cancelacion y tiempos limite en aplicaciones y scripts.

7.3 Control de flujo no local

Las funciones setjmp y longjmp permiten realizar saltos no locales cuando ocurre una señal. Son utiles para recuperar control desde contextos profundos del programa, aunque deben usarse con cuidado para evitar estados inconsistentes.

7.4 Codigo: Tratamiento de una señal (Ctrl+C)

Codigo Fuente (C)

Manejador personalizado para SIGINT con signal().

#include <stdio.h>
#include <signal.h>
#include <stdlib.h>
#include <unistd.h>

void manejador(int sig) {
    printf("Senal %d recibida. No morire tan facil.\n", sig);
}

int main() {
    // Establece la rutina de tratamiento
    if (signal(SIGINT, manejador) == SIG_ERR) exit(1);

    while (1) {
        printf("Esperando Ctrl+c...\n");
        sleep(2);
    }
    return 0;
}

Salida en terminal

Esperando Ctrl+c...
Esperando Ctrl+c...
Esperando Ctrl+c...
Senal 2 recibida. No morire tan facil.
Esperando Ctrl+c...
Terminado (killed)

El programa entra en un ciclo esperando interrupciones. Cuando llega SIGINT (2), el manejador imprime un mensaje y el proceso continua. Esto demuestra como una señal puede capturarse para modificar el comportamiento por defecto del programa.

Lo que aprendi y como se podria mejorar

Aprendi a interceptar SIGINT con un manejador para cambiar la respuesta por defecto y mantener el control del proceso.

Se puede mejorar migrando a sigaction() y agregando una rutina de cierre ordenado para liberar recursos.

7.5 Codigo: setjmp/longjmp con SIGUSR1

Codigo Fuente (C)

Salto no local con setjmp/longjmp al recibir SIGUSR1.

#include <stdio.h>
#include <signal.h>
#include <setjmp.h>
#include <stdlib.h>
#include <unistd.h>

jmp_buf env;

void sigusr1_handler(int sig) {
        signal(SIGUSR1, sigusr1_handler);
        longjmp(env, 1);
      }

      int main() {
        int i;

        signal(SIGUSR1, sigusr1_handler);

        for (i = 0; i < 10; i++) {
          if (setjmp(env) == 0)
            printf("Punto a regresar en el estado %d\n", i);
          else
            printf("Regreso al punto del estado %d\n", i);

          fflush(stdout);
          sleep(10);
        }

        return EXIT_SUCCESS;
}

Salida en terminal

Punto a regresar en el estado 0
Regreso al punto del estado 0

Este ejemplo muestra un salto no local: al iniciar, setjmp guarda un punto de retorno. Cuando llega SIGUSR1, el manejador ejecuta longjmp y la ejecucion regresa al bloque alterno del setjmp. Es util para recuperar control rapidamente ante eventos asincronos.

Lo que aprendi y como se podria mejorar

Aprendi como setjmp y longjmp permiten recuperar flujo de ejecucion despues de una señal asincrona en puntos de control definidos.

Se puede mejorar validando estados antes del salto y documentando las restricciones para evitar inconsistencias en variables compartidas.

7.6 Lo que aprendí

Aprendi que las señales son un mecanismo esencial de comunicacion asincrona entre el sistema y los procesos. Tambien comprendi la diferencia entre señales capturables y no capturables, y como un manejador puede cambiar la respuesta de una aplicacion ante eventos externos.

7.7 Cómo mejorarlo

Como mejora, se puede migrar de signal() a sigaction() para un control mas fino y portable, ademas de agregar manejo de SIGTERM/SIGALRM y limpieza segura de recursos en el manejador.
