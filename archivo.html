Practica #6

Arquitectura del Sistema de Archivos

Inodos Metadatos Bloques de Datos Jerarquia

Concepto general

En Linux, el sistema de archivos es logico y jerarquico. Se compone de cuatro secciones: Boot (arranque), Superbloque (metadatos del sistema), Lista de inodos (descripcion de archivos) y Bloque de datos. Esta estructura permite administrar eficientemente el almacenamiento y acceso a archivos.

6.1 Inodos

Cada archivo tiene un inodo con su informacion administrativa: propietario, permisos, tamaño, timestamps de creacion y modificacion. Sin embargo, el nombre del archivo no se almacena en el inodo, sino en el directorio que lo contiene.

6.2 Tipos de archivos

Linux reconoce varios tipos de archivos: regulares (datos), directorios (colecciones de archivos), de dispositivos en bloque o caracter (interfaces a hardware) y de comunicacion como tuberias (pipes) para transferencia de datos entre procesos.

6.3 Entrada/Salida (E/S)

Los controladores (drivers) gestionan la comunicacion con perifericos. El kernel abstrae los detalles hardware a traves de interfaces estandarizadas, permitiendo que las aplicaciones accedan a dispositivos de manera uniforme sin conocer sus particularidades tecnicas.

6.4 Codigo: Informacion del Sistema de Archivos

Codigo Fuente (C)

Uso de statvfs para obtener informacion estadistica del sistema de archivos.

#include <stdio.h>
#include <stdlib.h>
#include <errno.h>
#include <sys/types.h>
#include <sys/statvfs.h>
int main()
{
 struct statvfs vfs;
 char *ruta="/";
 if (statvfs(ruta, &vfs) != 0) {
 perror ("llamado de statvfs");
 exit(EXIT_FAILURE);
 }
 printf("\tArchivo:%s",ruta);
 printf("\tTamaño de bloques: %ld\n", (long) vfs.f_bsize);
 printf("\tTamaño de fragmento: %ld\n", (long) vfs.f_frsize);
 printf("\tTamaño en unidades: %lu\n", (unsigned long) vfs.f_blocks);
 printf("\tBloques libres %lu\n", (unsigned long) vfs.f_bfree);
 printf("\tBloques Disponibles: %lu\n", (unsigned long) vfs.f_bavail);
 printf("\tNúmero de Inodos: %lu\n", (unsigned long) vfs.f_files);
 printf("\tNúmero de Inodos Libres: %lu\n", (unsigned long) vfs.f_ffree);
 printf("\tNúmero de Inodos Disponibles: %lu\n", (unsigned long) vfs.f_favail);
 printf("\tID del S.A.: %#lx\n", (unsigned long) vfs.f_fsid);
 printf("\tBandera: ");
 if (vfs.f_flag == 0) printf("(Ninguna)\n");
 else {
if ((vfs.f_flag & ST_RDONLY) != 0) printf("ST_RDONLY ");
if ((vfs.f_flag & ST_NOSUID) != 0) printf("ST_NOSUID");
printf("\n");
}
 printf("\tLongitud max para archivo: %ld\n", (long)vfs.f_namemax);
 return EXIT_SUCCESS;
}

Salida en terminal

Archivo:/       Tamaño de bloques: 4096
Tamaño de fragmento: 4096
Tamaño en unidades: 51276028
Bloques libres 40982405
Bloques Disponibles: 38359519
Número de Inodos: 13099008
Número de Inodos Libres: 12444384
Número de Inodos Disponibles: 12444384
ID del S.A.: 0xd8cd624d29e45242
Bandera: 
Longitud max para archivo: 255

Este programa consulta el sistema de archivos raiz (/) usando statvfs y muestra informacion como el tamaño de bloques, bloques libres, numero total de inodos y su disponibilidad. Estos datos revelan estadisticas del sistema de archivos sin acceder a archivos individuales.

Lo que aprendi y como se podria mejorar

Aprendi a interpretar bloques e inodos disponibles para estimar la salud del sistema de archivos sin necesidad de herramientas adicionales.

Se puede mejorar calculando porcentajes de ocupacion y comparando varias rutas para identificar diferencias entre sistemas montados.

6.5 Codigo: Recuperar informacion de un Inodo

Codigo Fuente (C)

Uso de la funcion stat para obtener metadatos de un archivo.

#include <stdio.h>
#include <sys/stat.h>
#include <time.h>

int main(int argc, char *argv[]) {
    struct stat sb;
    if (stat(argv[1], &sb) == -1) return 1;

    printf("I-nodo: %ld\n", (long) sb.st_ino);
    printf("Tamaño: %lld bytes\n", (long long) sb.st_size);
    printf("Ultimo cambio: %s", ctime(&sb.st_ctime));
    return 0;
}

Salida en terminal

I-nodo: 11162033
Tamaño: 484 bytes
Ultimo cambio: Sat May 16 14:50:09 2026

Este programa consulta el inodo del archivo stat_inode.c y muestra su numero de inodo, tamaño en bytes y timestamp del ultimo cambio. Estos metadatos revelan informacion administrativa del archivo sin leer su contenido.

Lo que aprendi y como se podria mejorar

Aprendi que el inodo concentra metadatos esenciales del archivo y que su identificador permite rastrear objetos de forma precisa.

Se puede mejorar validando argumentos y mostrando mas campos como permisos, propietario, grupo y cantidad de enlaces.

6.6 Codigo: Informacion de Dispositivos

Codigo Fuente (C)

Uso de stat para identificar dispositivos y obtener numeros major/minor.

#include <stdio.h>
#include <stdlib.h>
#include <sys/types.h>
#include <sys/stat.h>
#include <unistd.h>
#include <sys/sysmacros.h> 
int main(int argc, char *argv[])
{
 struct stat sb;
 if (argc != 2) {
 fprintf(stderr, "Uso: %s <ruta_dispositivo>\n", argv[0]);
 exit(EXIT_FAILURE);
 }
 // stat obtiene información sobre el archivo de dispositivo
 if (stat(argv[1], &sb) == -1) {
 perror("stat");
 exit(EXIT_FAILURE);
 }
 // Verificar si es un dispositivo de bloques o caracteres
 printf("Archivo: %s\n", argv[1]);
 if (S_ISCHR(sb.st_mode)) {
 printf("Tipo: Dispositivo de caracteres (Char)\n");
 } else if (S_ISBLK(sb.st_mode)) {
 printf("Tipo: Dispositivo de bloques (Block)\n");
 } else {
 printf("Tipo: No es un dispositivo de bloques o caracteres\n");
 exit(EXIT_FAILURE);
 }
 // major() y minor() extraen los números correspondientes
 printf("Número Mayor (Major): %u\n", major(sb.st_rdev));
 printf("Número Menor (Minor): %u\n", minor(sb.st_rdev));
 return EXIT_SUCCESS;
}

Salida en terminal

Archivo: /dev/null
Tipo: Dispositivo de caracteres (Char)
Número Mayor (Major): 1
Número Menor (Minor): 3

Este programa examina el archivo especial /dev/null identificando que es un dispositivo de caracteres (Char) con numero mayor 1 y numero menor 3. Los numeros major y minor identifican de manera unica a cada dispositivo en el sistema, permitiendo al kernel dirigir operaciones al driver correcto.

Lo que aprendi y como se podria mejorar

Aprendi a distinguir dispositivos de bloque y caracter, y a interpretar major/minor para entender su relacion con el driver.

Se puede mejorar aceptando varias rutas en una sola ejecucion y mostrando errores mas claros para cada dispositivo evaluado.

6.7 Lo que aprendí

Aprendi a entender la separacion entre inodo y nombre de archivo, asi como la importancia de los metadatos en la administracion del sistema. Comprendi que el kernel abstrae la complejidad hardware y que la jerarquia del sistema de archivos es la clave para la organizacion logica del almacenamiento.

6.8 Cómo mejorarlo

Para mejorar esta practica se puede expandir el programa para mostrar mas metadatos como permisos en formato octal, propietario y grupo, cantidad de enlaces, y bloques asignados. Tambien es util comparar inodos de archivos en diferentes sistemas de archivos para entender variaciones de implementacion.
